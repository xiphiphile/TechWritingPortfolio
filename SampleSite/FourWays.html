One Process Four Ways

The examples below demonstrate four ways a single process could be designed, depending on how the automation is intended to work.

While the actions are the same, differences in the flow help to quickly demonstrate how the automation handles the process.

Exclusive Flow

In an exclusive flow, only one of the possible choices will run. A user would need to run the bot multiple times to run more than one option.

Looping Flow

In a looping process, the same basic process runs multiple times, with the number of runs governed by a list, spreadsheet, etc... Looping processes tend to be easier to view at a glance, but may not be as detailed as a sequence.

This is the most efficient flow option if the basic steps for multiple options will remain the same - you may not need to change the flow if new run options are added or deleted.

Sequential Flow

In a sequential process, the order of the processes run is laid out explicitly. It may be that the order is important for the automation to work, that the individual runs are not identical, or that the flow needs to contain details that would be lost if the steps were shown as a loop.

A loop running by defined rules can always be expressed as a sequence, but a sequence cannot always be expressed as a loop.

Looping Flow

In a concurrent flow, multiple subprocesses occur simultaneously. A concurrent process requires more resources at once, sometimes including multiple machines (physical or virtual), but would be more time efficient.

Some examples of when to use a concurrent flow would be:

  • Launcher bot initiates multiple subprocess bots designed to run at the same time.
  • Command sent to multiple external systems, which then report back to the host.
  • Bot network launches multiple bots od different machines.
